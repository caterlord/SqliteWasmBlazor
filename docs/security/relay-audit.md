# DeltaRelay Audit

_Generated by `/cryptosync-audit relay` against commit 94d0766. Source: skill SKILL.md "Mode 4: relay" + threat-model.md §2-§3._

Scope: metadata leakage and DoS / overload surface only. Payload confidentiality is out of scope (the relay sees opaque ciphertext from the worker's perspective).

---

## Critical

### C-1. POST endpoint is fully unauthenticated; allows unbounded inbox flooding

- **Severity:** Critical
- **Location:** `DeltaRelay/delta-relay.php:77-117`
- **Evidence:**

  ```php
  if ($method === 'POST') {
      $raw = file_get_contents('php://input');
      $req = json_decode($raw, true);
      ...
      foreach ($recipients as $pk) {
          ...
          $stmt->execute([':pk' => $pk, ':env' => $envelope, ':ts' => $now]);
      }
  ```

  The POST path performs no signature check, no rate-limit, no per-source accounting. Anyone with HTTPS reach to `/api/delta` can write a row for any recipient pubkey.
- **Threat:** This is the entry point for the revoked-member DoS scenario. A revoked or malicious actor retains their network access (no protocol-level way to revoke that), so they can keep posting envelopes to any inbox indefinitely. Recipients reject the garbage at the AEAD/Ed25519 boundary so confidentiality and integrity hold (P1, P4, P6 unaffected) — but bandwidth, receiver CPU, and relay disk are consumed without bound. Threat-model §3.6 lists this in scope.
- **Mitigation options** (operator chooses):
  1. **Bind POST to a sender signature** (e.g. `X-Sender-Sig` over `ts || sha256(envelope)`). Breaks the "no stored keys server-side" property because the relay would need to decide whose signature is valid (or accept any valid Ed25519 sig and use the pubkey from the request — which still doesn't bind to actor identity but does prove control of *some* keypair). Doesn't fully stop DoS but raises the cost from "anyone with curl" to "anyone willing to generate a keypair."
  2. **Rate-limit per source IP** at the web-server layer (nginx/Apache/Caddy). Operational, no protocol change.
  3. **Body-size cap + recipient fan-out cap** (see C-2, C-3). Independent of auth; bounds per-request damage.
  4. **Relay GC** (see C-4) so the bloat is reclaimable.

### C-2. No application-level body-size cap on POST

- **Severity:** Critical
- **Location:** `DeltaRelay/delta-relay.php:78`
- **Evidence:**

  ```php
  $raw = file_get_contents('php://input');
  $req = json_decode($raw, true);
  ```

  The script reads `php://input` whole, then base64-decodes. Only PHP's SAPI-level `post_max_size` (default 8 MB) limits this. There is no application-level check on either the raw body or the decoded envelope.
- **Threat:** Combined with the unbounded recipient fan-out (C-3), one 8 MB POST with 100 recipients writes ~800 MB to `relay.db`. With multiple concurrent attackers this saturates disk in minutes. Even without fan-out, attackers can sustain bandwidth-bound write rates trivially.
- **Mitigation:** Add an explicit cap, e.g. `if (strlen($raw) > 1 << 20) jsonOut(413, [...])` at the top of the POST handler. Reasonable cap is application-determined but should be ≤ 1 MB given V2 envelopes for normal sync deltas are small.

### C-3. No upper bound on recipient fan-out per POST

- **Severity:** Critical
- **Location:** `DeltaRelay/delta-relay.php:88-117`
- **Evidence:**

  ```php
  if (count($recipients) === 0) {
      jsonOut(400, ['error' => 'recipientPublicKeys is empty']);
  }
  ...
  foreach ($recipients as $pk) {
      if (!is_string($pk) || $pk === '') {
          throw new RuntimeException('recipient must be non-empty string');
      }
      $stmt->execute([':pk' => $pk, ':env' => $envelope, ':ts' => $now]);
  }
  ```

  The minimum-recipients check exists; there is no maximum. A single POST with `recipientPublicKeys` = an array of 10,000 entries triggers 10,000 INSERTs in one transaction.
- **Threat:** Amplification. One attacker request → N attacker rows for arbitrary N. Combined with C-1 (no auth) and C-2 (no body cap, only SAPI), this is a write-amplification DoS.
- **Mitigation:** Cap fan-out, e.g. `if (count($recipients) > 64) jsonOut(413, ['error' => 'too many recipients'])`. SyncEngine's recipient list is "every TrustedContact except self" (`SyncEngine.cs:236-244`) — typical group sizes are well under any reasonable cap.

### C-4. No retention GC; `relay.db` grows linearly forever

- **Severity:** Critical (operational)
- **Location:** `DeltaRelay/delta-relay.php:55-66`
- **Evidence:**

  ```php
  $pdo->exec(
      'CREATE TABLE IF NOT EXISTS deltas (
          cursor INTEGER PRIMARY KEY AUTOINCREMENT,
          recipient_pubkey TEXT NOT NULL,
          envelope BLOB NOT NULL,
          created_at INTEGER NOT NULL
      )'
  );
  ```

  No `DELETE FROM deltas WHERE …` exists anywhere in the file; `created_at` is recorded but never consulted.
- **Threat:** Even with C-1/C-2/C-3 mitigated, legitimate sync traffic accumulates. Memory `project_relay_design.md` open follow-up #5 documents the inverted-persistence GC design (relay requests fresh snapshot from a client, then `DELETE` deltas older than the snapshot's cursor). That design is not yet implemented.
- **Mitigation:** Two paths, in order of strength:
  1. **Inverted-persistence GC** as designed in the memory: `/api/snapshot` POST/GET endpoints + per-recipient `snapshot_requested_at` watermark + delete-after-snapshot.
  2. **Time-based interim retention** if the snapshot work is delayed: `DELETE FROM deltas WHERE created_at < strftime('%s','now') - 86400 * 30` (30-day retention) executed periodically. Coarser than the snapshot scheme; loses deltas a long-offline client hasn't pulled yet.

## Warning

### W-1. `index.php` leaks deployment fingerprint

- **Severity:** Warning
- **Location:** `DeltaRelay/index.php:5-6`
- **Evidence:**

  ```php
  echo "Directory: " . __DIR__ . "\n";
  echo "PHP Version: " . phpversion() . "\n";
  ```

  Anyone who hits the bare relay URL learns the absolute filesystem path on the host and the exact PHP version.
- **Threat:** Fingerprinting for targeted exploit selection. The PHP version exposes any unpatched CVEs at a glance; the directory exposes the deployment layout (potentially username, project name).
- **Mitigation:** Replace the body with a static string, e.g. `echo "<h1>Delta Relay</h1><p>API endpoint at /api/delta</p>";` and remove the `__DIR__` and `phpversion()` calls. Two-line edit.

### W-2. CORS open to any origin

- **Severity:** Warning
- **Location:** `DeltaRelay/delta-relay.php:27`
- **Evidence:**

  ```php
  header('Access-Control-Allow-Origin: *');
  ```

  Documented in memory `project_relay_design.md` as intentional (BlazorPRF-style "dumb store and CORS bypass"). Under the threat model the network attacker is already Dolev-Yao, so this doesn't widen the *integrity* surface — but it lets a malicious webpage launch the C-1 DoS through any visiting browser without CSRF mitigation.
- **Threat:** Cross-origin browser-based amplification of C-1. A drive-by webpage can flood the relay using each visitor's browser as an unwitting amplifier.
- **Mitigation:** Either pin `Access-Control-Allow-Origin` to the production CryptoSync app's origin (loses the BYO-transport flexibility), or document this as accepted policy and rely on C-1 mitigations (rate-limit at the web-server layer, body cap, fan-out cap) to bound damage. The latter is consistent with the threat model.

### W-3. PHP exception messages echoed verbatim on POST failure

- **Severity:** Warning
- **Location:** `DeltaRelay/delta-relay.php:113-116`
- **Evidence:**

  ```php
  } catch (Throwable $e) {
      $pdo->rollBack();
      jsonOut(400, ['error' => $e->getMessage()]);
  }
  ```

  Any PDO exception message is reflected to the client. PDO error messages can include SQL fragments and (depending on driver and `PDO::ATTR_ERRMODE`) bound parameter values.
- **Threat:** Information disclosure about the schema and possibly the fingerprinted environment. Lower priority than C-1/C-4 because the schema is documented and pubkeys posted by the attacker echoing back is not new information.
- **Mitigation:** Replace with a generic `jsonOut(400, ['error' => 'invalid request']);` and `error_log($e->getMessage())` for operator visibility.

## Info

### I-1. Default `X-Powered-By` and `Server:` headers may leak the PHP/web-server version

- **Severity:** Info
- **Location:** `DeltaRelay/delta-relay.php` + web-server config
- **Evidence:** No `header_remove('X-Powered-By')` call exists. Default PHP behavior emits `X-Powered-By: PHP/x.y.z` unless `expose_php=Off` in php.ini. Apache/Nginx adds its own `Server:` header.
- **Mitigation:** Operator-policy. Fix in php.ini and web-server config rather than in this file.

### I-2. Timing distinguishes "known recipient with N envelopes" from "unknown recipient"

- **Severity:** Info
- **Location:** `DeltaRelay/delta-relay.php:147-156`
- **Evidence:** The SELECT query and the response-body construction both scale with the number of envelopes returned. An attacker measuring response time learns approximately how many pending envelopes a target pubkey has.
- **Threat:** Pubkey is already wire-visible to any network attacker; envelope count is metadata under the threat model's stated routing-vs-secret classification (threat-model §8). Not a new surface.
- **Mitigation:** None recommended; the timing leak conveys the same information as the response body itself, both treated as routing metadata.

### I-3. No app-level logging — neither leaks linkable IDs nor records suspicious traffic

- **Severity:** Info (no leak), but flagged because it's a double-edged finding
- **Location:** Whole `delta-relay.php` — zero `error_log` / `file_put_contents` for app events.
- **Evidence:** No log statements in the file at all. Default PHP error logging may capture stack traces from `Throwable` paths but no app-level traffic is recorded.
- **Threat (positive):** No log-linkability leak (one of the metadata-leak checks the relay-mode audit looks for).
- **Threat (negative):** No telemetry to detect ongoing C-1/C-2/C-3 DoS abuse. Operator can't tell "this relay is being attacked" from disk fill alone.
- **Mitigation:** Add per-IP request-rate tracking (e.g. via redis or even a flat file) feeding back into a rate-limit decision. Couples with C-1 mitigation #2.

## Pre-existing safeguards (no finding, confirmation only)

- **Z-1.** Direct access to `relay.db` is denied by `LocalValetDriver.php:14-32`. SQLite file is not web-reachable.
- **Z-2.** GET endpoint requires Ed25519 signature verification *before* any I/O (`delta-relay.php:143` runs before SELECT at line 147). An attacker without a valid `(X-Timestamp, X-Sig)` cannot trigger the read path at all.
- **Z-3.** GET signature uses the recipient pubkey from the query as the verifier (`delta-relay.php:143` — `sodium_crypto_sign_verify_detached($sig, "$ts|$recipient", $pubKey)`). Stateless; no key registry, no account model.
- **Z-4.** `OPTIONS` preflight returns 204 immediately (`delta-relay.php:31-34`). No information disclosure.
- **Z-5.** No filenames-on-disk per envelope — single SQLite file. No filesystem enumeration leak.

## Summary

| C-1 | Critical | POST unauthenticated → revoked-member DoS, network-attacker DoS |
| C-2 | Critical | No application-level body-size cap |
| C-3 | Critical | No recipient fan-out cap |
| C-4 | Critical | No retention GC; `relay.db` grows forever |
| W-1 | Warning | `index.php` leaks `__DIR__` + PHP version |
| W-2 | Warning | CORS open to any origin |
| W-3 | Warning | PDO exception messages echoed on POST failure |

**4 Critical / 3 Warning / 3 Info.**

The C-1 cluster (C-1+C-2+C-3) is the single highest-priority finding because it's where the revoked-member DoS scenario actually executes. Mitigations are independent of each other and can land incrementally:

1. **Body-size cap (C-2)** — one-line change at the top of the POST handler. Lowest cost, immediate bound on per-request damage.
2. **Recipient fan-out cap (C-3)** — one-line change after the empty-array check. Same risk profile, same trivial fix.
3. **Rate-limit at web-server layer** — operational, no PHP change. Bounds attack rate per source IP.
4. **Retention GC (C-4)** — larger piece, designed already in memory `project_relay_design.md` (snapshot endpoints), independently scoped.

Sender-authenticated POST is *not* on this list because under the threat model the relay is honest-but-curious and stores no keys; binding POST to a signature would either require key registry server-side (breaks the architecture) or accept any Ed25519 sig (raises the bar but doesn't actually identify revoked members). The economically right defenses are rate-limiting + caps, not auth at the POST endpoint.
