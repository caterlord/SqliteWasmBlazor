## Confidentiality

### P1 -- Storage backend never observes plaintext payload bytes
- **Verdict:** Partial
- **Code:**
  - Worker at-rest: `SqliteWasmBlazor/TypeScript/worker/vfs-prf/sahpool-prf-vfs.ts:824-1053` (xRead/xWrite encrypt/decrypt every 4096-byte slot through `@sqlitewasmblazor/crypto-core`'s ChaCha20-Poly1305).
  - Per-row export: `SqliteWasmBlazor/TypeScript/worker/crypto-ops.ts:730` (`encryptAesGcm(rowBytes, cek, aad)`); the wire shape is `ShadowRow.EncryptedRow` (`SqliteWasmBlazor.CryptoSync/Models/DeltaEnvelope.cs:38-44`).
  - C# transport: `SqliteWasmBlazor.CryptoSync/Services/HttpSyncTransport.cs:47-74` posts the already-MessagePacked envelope verbatim; relay handler `DeltaRelay/delta-relay.php:77-117` writes it as an opaque BLOB.
- **Tests:**
  - `SqliteWasmBlazor.CryptoSync.Tests/PrfVfsEnvelopeTests.cs:69` (BC vs noble round-trip + tamper)
  - `SqliteWasmBlazor.CryptoSync.Tests/InvitationRoundtripTests.cs:102` (`RespondToInvitation_WireOpacity_NoPlaintextLeak` — substring scan over the wire)
  - `SqliteWasmBlazor/TypeScript/worker/vfs-prf/__tests__/envelope.test.ts:90`
- **Notes:** Plaintext-substring scan exists for the invitation channel only. No equivalent assertion runs over a synthesized `DeltaEnvelope` body or over the relay's stored BLOB. Test gap, not a policy gap. PBT recommendation: extend `AssertNoPlaintextLeak` (`InvitationRoundtripTests.cs:303`) into a `fast-check`-driven property over random plaintexts → encrypt → scan envelope bytes for any 8-byte plaintext substring.

### P2 -- Secret metadata encrypted under same/stronger key, AAD-bound
- **Verdict:** Covered (by stated policy)
- **Code:**
  - Metadata classification per `docs/security/threat-model.md` §8: row payloads secret (in `EncryptedRow` body); recipient/sender pubkey, `groupContext`, `keyVersion`, `SharingScope`, `SharingId`, row `Id`, envelope size, request timestamp are routing metadata — visible to the relay by design.
  - Worker AAD: `crypto-ops.ts:150-152` (`buildAad` = `${groupContext}:${keyVersion}`).
  - VFS at-rest: `SqliteWasmBlazor/TypeScript/worker/vfs-prf/aad.ts:17-26` (`prf-vfs-v1|dbPath|slotIndex`).
- **Tests:** `V2CryptoHeaderTests.cs:91` covers the AAD string shape; `envelope.test.ts:141` covers cross-DB swap rejection.
- **Notes:** Threat model commits the metadata classification. What was previously "Out of scope pending decision" is now stated policy: routing fields are not encrypted, row bodies are. Adversary-graph leakage from routing fields (group sizes, sync frequency, recipient relationships) is acknowledged as a metadata trade-off in the threat model.

### P3 -- IND-CPA at storage layer (equal-length payloads indistinguishable)
- **Verdict:** Partial

## Integrity and Authentication

### P4 -- Modification of stored ciphertext or metadata detected on read
- **Verdict:** Covered
- **Code:**
  - At-rest: `sahpool-prf-vfs.ts:962-967` decrypt path raises through `decryptChaCha20Poly1305`'s tag verification; `sahpool-prf-vfs.ts:561-595` (`verifyEncryptionKey`) probes slot 0 explicitly.
  - Wire: `crypto-ops.ts:911-918` (per-group batch signature verification rejects entire group on tamper); `crypto-ops.ts:931-943` (per-row AEAD decrypt failure → `TAMPER_AAD_MISMATCH`).
- **Tests:**
  - `envelope.test.ts:153` (bit-flip rejection)
  - `PrfVfsEnvelopeTests.cs:148` (`PageEnvelope_TamperedCiphertext_DecryptReturnsNull`)
  - `DeclarationSignerTests.cs:43, 54, 91, 117, 144, 172` (canonical-payload tampering rejected)
- **Notes:** Solid. Two layers (per-row AEAD + per-group batch Ed25519 sig + outer envelope sig). Relay-side hex-storage path is a `BLOB` column so byte-flips are stored verbatim and detected only on the receiver — that is the correct design for a dumb relay.

### P5 -- AAD binds ciphertext to logical identifier, version, owning context
- **Verdict:** Covered
- **Code:**
  - VFS slot AAD: `vfs-prf/aad.ts:17-26` (`prf-vfs-v1|dbPath|slotIndex_LE32`).
  - Per-row AAD: `crypto-ops.ts:150-152` (`groupContext:keyVersion`).
  - Outer envelope sig binds the `pack(groups)` byte sequence: `crypto-ops.ts:818-825`.
- **Tests:**
  - `envelope.test.ts:141` (`decrypt fails with the wrong AAD`)
  - `envelope.test.ts:181-258` (slot-swap detection across slot indices and dbPaths)
  - `PrfVfsEnvelopeTests.cs:113, 132` (cross-DB and cross-slot-index AEAD failure)
  - `rekey.test.ts:159` (`rekeyed output decrypts only under the matching dbPath AAD`)
- **Notes:** Per-row AAD is `${groupContext}:${keyVersion}` — does **not** bind `Id`, `SharingId`, or `tableName`. The per-group batch Ed25519 signature is what actually prevents row-relocation between tables (the signature covers the row's ciphertext+nonce, and the wire `tableName`/`isSystemTable` is signed via the outer signature over `pack(groups)`). Open question logged in threat-model §9: should the AAD extend to bind `tableName`/`Id` for defense in depth? PBT recommendation: lock the swap-rejection assumption explicitly with `aad-swap-rejection` (recipe in `pbt-recipes.md`).

### P6 -- Only clients with the right subkey produce accepted ciphertexts
- **Verdict:** Covered (by stated policy)
- **Code:**
  - Receive-side sender check: `crypto-ops.ts:907-918` (batch Ed25519 verify); `crypto-ops.ts:984-1000` (system-table sender must be the local-DB admin); `crypto-ops.ts:421-520` (domain-table sender role lookup gated on a chain of signatures: ShareTarget AdminSignature, GroupAdmin trust check, PermissionTableSignature).
  - Group-control plane: `GroupService.cs` (`CreateGroupAsync`, `AddMembersAsync`, `RemoveMemberAsync`) takes `DualKeyPairFull` and signs every new `ShareTarget` with the admin's Ed25519 priv via `DeclarationSigner.SignShareTargetAsync`; `GroupTransferService.cs:264-289` already followed this pattern.
  - Relay-side write auth: `DeltaRelay/delta-relay.php:77-117` — POST endpoint is **unauthenticated** (anyone with HTTPS can write to any recipient pubkey's inbox). GET path verifies `sodium_crypto_sign_verify_detached($sig, "$ts|$recipient", $pubKey)` — recipient pubkey is the verifier (`delta-relay.php:143`).
- **Tests:** `DeclarationSignerTests.cs` (canonical signature shapes); `SignedShareTargetTests.cs:127-178` (tampering rejected); `GroupServiceTests.cs:CreateGroup_NewTargetsCarryValidAdminSignature`, `AddMembers_NewTargetsCarryValidAdminSignature`, `RemoveMember_NewTargetsCarryValidAdminSignature` (every `GroupService` write path produces a credential the admin's Ed25519 pubkey verifies); `SyncEngineTests.cs:54` (recipient enumeration).
- **Notes:** **Anonymous POST to the relay** remains an acknowledged DoS surface; not a confidentiality/integrity break. Mitigations are operator-policy (rate-limit, body-size cap) and tracked in the relay-mode audit, not here. **Phase 1 remediation closed 2026-04-28**: `GroupService` was previously creating `ShareTarget` rows without `AdminSignature` / `GroupAdminEd25519PublicKey` — fixed by migrating all three write paths to take `DualKeyPairFull` and sign each credential.

## Replay and Freshness

### P7 -- Previously valid ciphertext cannot be substituted for newer one in same logical slot
- **Verdict:** Covered (by stated policy)
- **Code:**
  - Receiver side: `HttpSyncTransport.cs:88-142` advances `_cachedCursor` strictly forward. The relay's `cursor` column is `INTEGER PRIMARY KEY AUTOINCREMENT` (`delta-relay.php:56-58`), so an attacker cannot make a stored row appear at a lower cursor.
  - Receiver does NOT track per-(sender, slot-id) versions; under DeltaWins (`bulk-ops.ts:59`, `crypto-ops.ts:1160-1163`) an attacker who controls a sender's pubkey can re-deliver an old `EncryptedRow` for the same `Id` and the receiver will overwrite.
- **Tests:** `HttpSyncTransportTests.cs:122-160` (cursor advancement). No "sender re-sends old envelope at higher cursor" PBT.
- **Notes:** Threat model §8 commits **DeltaWins as explicit policy** (last-arriving-at-receiver wins by relay cursor). The literal "cannot be substituted" property text is intentionally weakened by this policy; the policy of record is "newest-arriving wins, ordered by relay cursor." PBT recommendation: write `Properties/StaleReplayProperty.cs` to **lock the policy as documented** (i.e. assert the open-table row reflects the most-recently-arrived envelope, regardless of inner `UpdatedAt`). Future column-level merge work would reverse this.

### P8 -- Auth tokens / signed requests include nonce or monotonic counter; replays rejected
- **Verdict:** Covered (by stated policy)
- **Code:** `delta-relay.php:127-145` — GET auth verifies Ed25519 signature over `"$ts|$recipient"` with `abs(time() - (int)$ts) > 300` clock-skew rejection. **No per-pubkey monotonic counter, no nonce store.**
- **Tests:** `HttpSyncTransportTests.cs:163-189` exercises the signing format.
- **Notes:** Threat model §8 commits **replay-within-300s as acceptable**: a captured `(X-Timestamp, X-Sig)` lets an attacker refetch the same envelopes the legitimate client could fetch, all of which are AEAD-protected and addressed to that pubkey. The leaked information is bounded to "same content the legitimate client could fetch." Future tightening (per-pubkey nonce table at relay) is tracked as Phase 4 in the remediation plan; not blocking under current policy.

### P9 -- Sync rounds advance monotonically; rollback to earlier snapshot detectable
- **Verdict:** Covered
- **Code:**
  - Receive cursor: `HttpSyncTransport.cs:43-141` + `IReceiveCursorStore.cs`. Default `InMemoryReceiveCursorStore` for tests/dev; production injects `EfReceiveCursorStore` (`Services/EfReceiveCursorStore.cs`) which persists to `SyncState.LastReceivedCursor` on the deterministic-id row, sharing the row with the export cursor. Durability comes from the OPFS-hosted SQLite the rest of the app already uses.
  - Export cursor: `SyncEngine.cs:36-145` persists `LastExportedAt` to the same `SyncStates` row (`Models/SyncState.cs`).
- **Tests:** `SyncEngineTests.cs:120` (engine-side persist + restore); `HttpSyncTransportTests.cs:206-245` (cursor-store persist + resume against the seam); `EfReceiveCursorStoreTests.cs` (durability across context instances, both-cursors-on-same-row non-clobber invariant).
- **Notes:** **Phase 2 remediation closed 2026-04-28**: receive cursor is now durable through reload. Cursor and export cursor share the deterministic-id `SyncState` row, so a single fetch-or-create covers both columns; concurrent writes degrade to last-writer-wins on whichever column raced (cursors only ever advance, so a lost write at most replays a small batch). Forward-compatible with the pending snapshot/watermark GC mechanism (memory `project_relay_design.md` open follow-up #5) — when `/api/snapshot` ships, the snapshot endpoint will seed this same cursor on cold start.

## Key Hygiene

### P10 -- Each derived key used in exactly one cryptographic context, established by domain-separated KDF
- **Verdict:** Covered
- **Code:**
  - Wrapping-key derivation: `crypto-ops.ts:134-148` (`unwrapCekFromHeader` → `deriveWrappingKey(myPriv, peerPub, groupContext)`); `groupContext` is the HKDF info parameter.
  - PRF-derived domain keys: `SqliteWasmBlazor.Crypto/Services/PrfService.cs:213-260` uses `HKDF.DeriveKey(SHA256, ikm: seed, output: 32, salt: empty, info: contextBytes)` — distinct contexts produce distinct keys.
  - Threat-model §6 captures the full key hierarchy (PRF seed → identity X25519/Ed25519, ECIES → wrapping key, per-group CEK, per-DB VFS key) with disjoint derivation paths.
- **Tests:** `PrfVfsEnvelopeTests.cs:55-67` (AAD format); rekey tests in `rekey.test.ts:94-153` and `PrfVfsEnvelopeTests.cs:267-393` round-trip through HKDF→content key→encrypt without context collision.
- **Notes:** No explicit PBT that "for distinct context strings, derived keys differ; for equal contexts, equal key" — recommend adding `kdf-context-separation` to lock the invariant against accidental refactor.

### P11 -- Nonces unique across all clients and all keys
- **Verdict:** Partial
- **Code:**
  - VFS write: `sahpool-prf-vfs.ts:1023` (random nonce per `encryptChaCha20Poly1305` call via crypto-core's `generateRandomBytes(12)`).
  - Per-row AES-GCM: `aesGcm.ts:26` (`generateRandomBytes(NONCE_LENGTH_AES)`).
  - **No counter-based scheme; no persistence.** Both paths rely on `crypto.getRandomValues()` over a 12-byte nonce.
- **Tests:** `envelope.test.ts:170` ("two encryptions of identical plaintext produce different ciphertexts" — single-process check). No PBT for collision-bound across simulated clients.
- **Notes:** With a 96-bit random nonce and AES-GCM, the birthday bound (~2^32 messages per key for ε=2^-32 collision probability) is ample for normal use. The threat-model risk is at scale: a long-lived CEK that encrypts >2^32 rows could collide. **No code path enforces a per-CEK message count cap or a key-rotation cadence keyed off message count.** Threat-model §9 logs the policy question (per-CEK cap vs. operational cadence) as open. PBT for `nonce-uniqueness` over N simulated clients × M operations is the right shape to lock the assumption.

### P12 -- Key rotation produces new key with no algebraic relation to previous
- **Verdict:** Covered
- **Code:**
  - Group rotation: `GroupService.cs:158-193` and `GroupTransferService.cs:231-302` — both call `groupEncryption.RotateGroupKeyAsync(...)`. The rotated CEK is freshly generated by the underlying provider (no derivation from old CEK).
  - Slot rekey primitive: `vfs-prf/rekey.ts:28-92` (decrypt under K_old, re-encrypt under caller-supplied K_new — keys are unrelated except for caller's choice).
- **Tests:** `rekey.test.ts:94-197`; `PrfVfsEnvelopeTests.cs:267-393`; `GroupTransferServiceTests.cs:163-197` (`FullTransfer_NewAdminCanUnwrapRotatedCek`).
- **Notes:** Memory `project_vfs_export_rekey.md` (locked 2026-04-26) matches the implementation under `vfs-prf/rekey.ts`. Old ciphertexts remain decryptable by whoever still holds K_old; "decryptable only per policy" is up to the host (no hardware-rooted forward-secrecy claim is made).

## Concurrency and Conflict

### P13 -- Two clients writing concurrently produce a deterministic conflict state
- **Verdict:** Covered (by stated policy)
- **Code:** `bulk-ops.ts:59` (DeltaWins overwrite); `crypto-ops.ts:1160-1163` (delta import always passes `mode=3`).
- **Tests:** `TwoActorFixtureTests.cs` covers post-bootstrap state.
- **Notes:** Threat model §8 commits **DeltaWins as the policy of record**: receiver applies envelopes in relay-cursor order; last-arriving wins. Deterministic given the relay-cursor total order. Future column-level merge with permission gating (per memory `project_conflict_resolution.md`) would change this — file an updated threat model when it lands.

### P14 -- Conflict resolution does NOT produce merged plaintext from inputs the resolver was not authorized to read
- **Verdict:** Partial
- **Code:** `crypto-ops.ts:1009-1097` — the import path either accepts an entire row (subject to permission gating per column at `crypto-ops.ts:1079-1092`) or rejects it. **No cross-row merge happens.** Permission-denied rows are not written to either the open or the shadow table (see comment at `crypto-ops.ts:1099-1101`), satisfying the `feedback_denied_rows_no_shadow.md` constraint.
- **Tests:** `SharingServiceTests.cs:42-93` (cascade soft-delete isolation); `crypto-ops.ts` permission-enforcement is exercised end-to-end in the Demo run only.
- **Notes:** The denied-rows-no-shadow invariant is enforced in code but the **C# unit-test suite does not exercise the permission-denied path**. Test gap, not a policy gap. **Fix planned in remediation Phase 3** — `Properties/PermissionDeniedNoShadowProperty.cs` will lock the invariant.

### P15 -- Stale client's write does not silently overwrite a newer write
- **Verdict:** Covered (by stated policy)
- **Code:** Same as P13. `EncryptedRow` body carries `UpdatedAt` (inside the AEAD), but the import path's `bulkInsertRows(..., mode=3)` does not compare `UpdatedAt` before overwriting — see `crypto-ops.ts:1160-1163`.
- **Tests:** None specific.
- **Notes:** Threat model §8 commits **last-arriving-at-receiver wins** as the explicit policy. A late envelope from a slow client overwrites a faster client's later write — acknowledged trade-off under DeltaWins. Property satisfied because the policy is now of record; future column-level merge would require a new threat-model entry.

## Recovery and Compromise

### P16 -- Compromised device reads only its own history, not future content of other devices
- **Verdict:** Covered (by stated policy)
- **Code:**
  - `LeaveService.cs:34-91` soft-deletes the leaver's own `ShareTarget` and persists a `LeaveDeclaration`. Cooperative — does not revoke key material on its own.
  - `GroupService.RemoveMemberAsync` (`GroupService.cs:133-194`) does call `RotateGroupKeyAsync` and re-wraps for remaining members.
  - `vfs-prf/rekey.ts` covers DB-level rekey for at-rest material.
- **Tests:** `LeaveServiceTests.cs:30-115` (declaration + soft-delete + signature roundtrip); `GroupTransferServiceTests.cs:163-197` (transfer + rotation + new-admin can unwrap CEK).
- **Notes:** Threat model §8 commits **revocation effective only after the next manual key rotation**. Under this policy: compromised device reads its own history (anything it could decrypt before compromise) and any envelope at the pre-rotation `KeyVersion` until admin rotates. The application *should* prompt the admin to rotate on observing a leave/revocation. PBT recommendation (`Properties/LeaveRotationForwardSecrecyProperty.cs`): leave → admin-rotate → assert leaver-keys-fail-to-decrypt the post-rotation envelope. This locks the policy.

### P17 -- Loss of device recoverable without re-encrypting all historical data
- **Verdict:** Covered (by stated policy)
- **Code:** Invitation re-flow at `ContactInvitationService.cs:CreateInvitationAsync` / `RespondToInvitationAsync` / `IngestInvitationResponsesAsync` / `PromoteInvitationAsync` (per memory `project_invitation_flow.md`); WebAuthn PRF re-derivation: `Crypto/Services/PrfService.cs:DeriveKeysAsync`.
- **Tests:** `InvitationRoundtripTests.cs:177-261` (`FullE2E_CreateRespondIngestPromote_BindsTrustedContactAndDeletesChannel`).
- **Notes:** Threat model §8 commits **re-encryption on grant: future content only** as policy. New device gets fresh `ShareTarget` rows for current `KeyVersion`s only; historical rows under retired keys are not re-shipped or re-encrypted. The trade-off (a brand-new device cannot recover state predating its `ShareTarget`) is acknowledged.

### P18 -- Documented revocation effective at protocol level, not just server ACL
- **Verdict:** Covered (by stated policy)
- **Code:**
  - Revocation declaration: `DeclarationSigner.cs:135-167`. Admin-override transfer: `DeclarationSigner.cs:172-203`.
  - **No code path automatically rotates group keys when a `RevocationDeclaration` is observed.** The relay (`delta-relay.php`) has no notion of revocation.
- **Tests:** `DeclarationSignerTests.cs:132-181` (signature shapes only).
- **Notes:** Threat model §8 commits **revocation effective only after manual rotation**. Revocation is signed and propagated; enforcement is the admin running `GroupService.RemoveMemberAsync` (or `GroupTransferService.ClaimGroupAsync` in the override case). The relay-side ACL is intentionally absent — relay is honest-but-curious, cannot be trusted to enforce ACLs. Future tightening (auto-rotation on declaration import) is tracked as Phase 5 in the remediation plan; not blocking under current policy.
